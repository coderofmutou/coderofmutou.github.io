# Maven 依赖管理与项目构建

## 一、概念与定位：Maven 解决的问题

Maven 的核心价值在两件事：

1. **依赖管理**：自动下载、传递依赖、版本统一、冲突裁决
2. **构建管理**：统一编译、测试、打包、安装、部署流程

Maven 可视作"**项目构建与依赖的标准操作系统**"。

::: tip
在团队协作中，Maven 真正的价值不是"方便下载 jar"，而是"让每个人、每台机器、每条 CI 流水线执行相同构建动作并得到可复现结果"。
:::

## 二、环境准备与基础配置（Windows）

### 前置条件

- JDK 17+（建议长期支持版本）
- 正确设置 `JAVA_HOME`
- `mvn -v` 能输出版本信息

### 建议检查项

```bash
mvn -v
```

期望输出至少包含：

- Apache Maven 版本
- Java 版本
- 操作系统信息

### settings.xml 常见配置

通常在 Maven 安装目录下 `conf/settings.xml` 或用户目录下 `.m2/settings.xml` 中进行配置。

#### 本地仓库路径

```xml
<localRepository>D:/maven-repository</localRepository>
```

#### 镜像配置（示例）

```xml
<mirrors>
    <mirror>
        <id>aliyunmaven</id>
        <name>aliyun maven</name>
        <url>https://maven.aliyun.com/repository/public</url>
        <mirrorOf>central</mirrorOf>
    </mirror>
</mirrors>
```

::: warning
`settings.xml` 里的 `servers` 可能包含私服账号密码，不要提交到 Git 仓库。
:::

## 三、Maven Wrapper（mvnw）

Maven Wrapper（`mvnw`）的作用是**在项目内锁定 Maven 版本**，确保任何人/任何机器/任何 CI 环境都使用相同的 Maven 版本构建，无需机器预先安装指定版本。

### 生成 Wrapper

```bash
# 在项目根目录下执行（需要本机已有 Maven）
mvn wrapper:wrapper -Dmaven=3.9.9
```

生成后的文件结构：

```text
demo-maven/
├── .mvn/
│   └── wrapper/
│       └── maven-wrapper.properties   # 指定 Maven 版本与下载地址
├── mvnw                               # Unix/Linux/macOS 执行脚本
├── mvnw.cmd                           # Windows 执行脚本
└── pom.xml
```

建议将以上所有文件 **纳入版本控制**（`.mvn/wrapper/maven-wrapper.jar` 在新版本中已默认不再生成，无需关注）。

### 统一使用 Wrapper

```bash
# 替代全局 mvn 命令
./mvnw clean verify          # Linux/macOS
.\mvnw.cmd clean verify      # Windows PowerShell
```

### 更新 Wrapper 版本

将 `-Dmaven=X.Y.Z` 中的版本号替换为目标版本，重新执行即可。

```bash
mvn wrapper:wrapper -Dmaven=3.9.9
```

::: tip
CI 流水线中优先使用 `./mvnw` 而不是 `mvn`，保持与本地开发环境的版本一致性。
:::

## 四、最小工程实践：从创建到运行

### 项目坐标（GAV）

Maven 项目由三个核心标识组成：

- `groupId`：组织/公司维度
- `artifactId`：模块名称
- `version`：版本号

完整坐标形式：`groupId:artifactId:version`

### GAV 命名约定（团队级）

在多人协作场景下，建议约定 GAV 命名规则，降低仓库污染与依赖歧义：

- `groupId`：建议使用反域名结构，如 `com.company.business`
- `artifactId`：建议使用"产品线-模块"语义，如 `order-service-api`
- `version`：建议遵循 `主版本.次版本.修订号`（如 `1.2.3`）

::: tip
命名规则的核心目标是"可检索、可区分、可演进"，而不是追求形式复杂度。
:::

### 最小可用 POM

```xml
<project xmlns="http://maven.apache.org/POM/4.0.0"
                 xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
                 xsi:schemaLocation="http://maven.apache.org/POM/4.0.0 https://maven.apache.org/xsd/maven-4.0.0.xsd">

        <modelVersion>4.0.0</modelVersion>

        <groupId>com.example</groupId>
        <artifactId>demo-maven</artifactId>
        <version>1.0.0-SNAPSHOT</version>

        <properties>
                <maven.compiler.release>17</maven.compiler.release>
                <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
                <project.reporting.outputEncoding>UTF-8</project.reporting.outputEncoding>
        </properties>

</project>
```

### 标准目录结构（强烈建议遵循）

```text
demo-maven/
├── pom.xml
└── src
        ├── main
        │   ├── java
        │   └── resources
        └── test
                ├── java
                └── resources
```

## 五、核心机制：生命周期与命令

Maven 内置三套生命周期：

- `clean`
- `default`
- `site`

日常开发最常用的是 `default` 生命周期中的这几步：

| 阶段 | 作用 |
|------|------|
| `compile` | 编译主代码 |
| `test` | 运行单元测试 |
| `package` | 打包（jar/war） |
| `verify` | 验证（含额外检查） |
| `install` | 安装到本地仓库 |
| `deploy` | 部署到远程仓库 |

### 常用命令组合

```bash
mvn clean
mvn clean test
mvn clean package
mvn clean install
mvn clean deploy
```

### 生命周期、命令、插件的关系

Maven 构建过程可以理解为三层映射：

1. 生命周期（Lifecycle）定义阶段序列
2. 命令（Phase）触发对应阶段及其前置阶段
3. 插件（Plugin Goal）最终完成实际动作

可简化为：`Lifecycle -> Phase -> Plugin Goal`。

例如执行 `mvn package` 时，本质上会触发 `package` 及其前置阶段，
并由编译、测试、打包相关插件完成具体工作。

::: tip
执行任意阶段会自动触发其所有前置阶段。
:::

## 六、核心机制：POM 关键结构块

### packaging

- `jar`：默认值，普通 Java 项目
- `war`：Web 项目
- `pom`：父工程/聚合工程

### dependencies 与 scope

```xml
<dependencies>
    <dependency>
        <groupId>org.junit.jupiter</groupId>
        <artifactId>junit-jupiter-api</artifactId>
        <version>5.11.0</version>
        <scope>test</scope>
    </dependency>
</dependencies>
```

常见作用域：

- `compile`：默认，编译/测试/运行都可见
- `test`：仅测试可见
- `provided`：编译与测试可见，运行由容器提供
- `runtime`：运行与测试可见，编译不可见

补充作用域（高频于工程治理）：

- `system`：与 `provided` 类似，但依赖来自本机路径；可移植性差，通常不建议使用
- `import`：仅用于在 `dependencyManagement` 中导入 BOM（`type=pom`）

### properties（统一变量）

```xml
<properties>
    <spring.version>6.1.10</spring.version>
    <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
</properties>
```

### build / plugins（构建行为定制）

```xml
<build>
    <plugins>
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-compiler-plugin</artifactId>
            <version>3.13.0</version>
            <configuration>
                <release>17</release>
            </configuration>
        </plugin>
    </plugins>
</build>
```

## 七、工程实践：依赖治理（传递、冲突、BOM）

### 传递依赖

如果 A 依赖 B，B 依赖 C，那么 A 默认会拿到 C（传递依赖）。

该机制提升了复用效率，但也可能导致依赖树快速膨胀。

### 冲突裁决规则

Maven 默认优先：

1. 路径更短的版本（nearest）
2. 路径同深度时，先声明优先

### 用 dependencyManagement 统一版本

```xml
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-dependencies</artifactId>
            <version>3.3.1</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>
```

子模块引用时可省略版本号。

### 排除不需要的传递依赖

```xml
<dependency>
    <groupId>com.example</groupId>
    <artifactId>demo-core</artifactId>
    <version>1.0.0</version>
    <exclusions>
        <exclusion>
            <groupId>commons-logging</groupId>
            <artifactId>commons-logging</artifactId>
        </exclusion>
    </exclusions>
</dependency>
```

::: warning
排除依赖前应先查看依赖树，避免运行期出现类缺失问题。
:::

### 传递终止：`optional`

当某依赖仅在当前模块内部需要、且不希望继续向下游模块传递时，可使用 `optional=true`：

```xml
<dependency>
    <groupId>com.alibaba</groupId>
    <artifactId>druid</artifactId>
    <version>1.2.15</version>
    <optional>true</optional>
</dependency>
```

该配置会阻断该依赖的默认传递，使用方需要时应显式声明。

## 八、工程实践：Profiles 构建环境切换

Profiles 让同一套 POM 在不同环境下有不同构建行为，适合开发/测试/生产的差异化配置。

### 基础示例

```xml
<profiles>
    <profile>
        <id>dev</id>
        <activation>
            <activeByDefault>true</activeByDefault>
        </activation>
        <properties>
            <env>dev</env>
            <log.level>DEBUG</log.level>
        </properties>
    </profile>

    <profile>
        <id>prod</id>
        <properties>
            <env>prod</env>
            <log.level>WARN</log.level>
        </properties>
        <build>
            <plugins>
                <!-- 生产环境额外生成 javadoc 包 -->
                <plugin>
                    <groupId>org.apache.maven.plugins</groupId>
                    <artifactId>maven-javadoc-plugin</artifactId>
                    <executions>
                        <execution>
                            <goals><goal>jar</goal></goals>
                        </execution>
                    </executions>
                </plugin>
            </plugins>
        </build>
    </profile>
</profiles>
```

### 激活方式

```bash
# 命令行显式激活
mvn clean package -Pprod

# 通过系统属性触发（需在 activation 中配置 property）
mvn clean package -Denv=prod

# 查看当前激活的 profile
mvn help:active-profiles
```

### Profile 定义位置与用途

| 定义位置 | 适用场景 |
|----------|----------|
| `pom.xml`（`<profiles>`） | 工程级差异，可移植，推荐优先 |
| 用户目录 `.m2/settings.xml` | 机器级，适合包含账号密码等本机信息 |
| 全局 `conf/settings.xml` | 全局默认，适合统一镜像/代理配置 |

::: tip
Profile 中的配置是对基础 POM 的 **叠加** 而非替换。激活多个 Profile 时，各 Profile 中的配置会依次合并。
:::

::: warning
避免在 Profile 里做"大换血"——例如整体替换依赖列表——会导致构建行为难以追踪。Profile 的粒度越细越好。
:::

## 九、工程实践：插件治理

推荐在父工程通过 `pluginManagement` 锁定插件版本：

```xml
<build>
    <pluginManagement>
        <plugins>
            <plugin>
                <groupId>org.apache.maven.plugins</groupId>
                <artifactId>maven-compiler-plugin</artifactId>
                <version>3.13.0</version>
            </plugin>
            <plugin>
                <groupId>org.apache.maven.plugins</groupId>
                <artifactId>maven-surefire-plugin</artifactId>
                <version>3.2.5</version>
            </plugin>
        </plugins>
    </pluginManagement>
</build>
```

这样可以避免不同机器默认插件版本不一致。

### 常用插件速查

| 插件 | 作用 | 典型配置项 |
|------|------|-----------|
| `maven-compiler-plugin` | 设置编译 source/target/release | `<release>17</release>` |
| `maven-surefire-plugin` | 运行单元测试 | 指定包含/排除测试类 |
| `maven-failsafe-plugin` | 运行集成测试 | 绑定 `integration-test` 阶段 |
| `maven-source-plugin` | 打包源码 | 绑定 `package` 阶段生成 `-sources.jar` |
| `maven-resources-plugin` | 处理 `src/main/resources` 资源文件，控制变量替换（filtering） | 见下方"资源打包补充" |
| `maven-dependency-plugin` | 提供 `copy-dependencies` 等目标，复制依赖 jar 到指定目录 | `<outputDirectory>` |
| `maven-shade-plugin` | 将依赖解压合并进单一可执行 jar（uber jar），支持按策略合并同名资源 | `<transformers>` |
| `maven-assembly-plugin` | 按描述符（内置或自定义）打包归档，可同时产出多种格式 | `<descriptorRefs>` / `<descriptor>` |
| `maven-javadoc-plugin` | 打包 API 文档 | 生成 `-javadoc.jar` |
| `maven-checkstyle-plugin` | 代码风格检查 | 指定 checkstyle 规则文件 |
| `spotbugs-maven-plugin` | 静态缺陷扫描 | 绑定 `verify` 阶段 |

示例：同时生成源码包与文档包。

```xml
<build>
    <plugins>
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-source-plugin</artifactId>
            <version>3.3.1</version>
            <executions>
                <execution>
                    <id>attach-sources</id>
                    <goals>
                        <goal>jar-no-fork</goal>
                    </goals>
                </execution>
            </executions>
        </plugin>
    </plugins>
</build>
```

::: tip
插件版本建议在父工程通过 `pluginManagement` 统一锁定，而不是在每个子模块重复写版本号。
:::

### 打包合并依赖：uber jar 与自定义 assembly 描述符

当需要产出一个"依赖全部合并在一起"的单一可执行 jar（通常称为 uber jar / fat jar，注意这与 Spring Boot 的可执行 jar 结构不同，后者是嵌套读取而非解压合并），常用 `maven-shade-plugin`：

```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-shade-plugin</artifactId>
    <version>3.5.1</version>
    <executions>
        <execution>
            <phase>package</phase>
            <goals><goal>shade</goal></goals>
            <configuration>
                <transformers>
                    <!-- 按行合并 META-INF/services/*，而非让后写入的覆盖先写入的 -->
                    <transformer implementation="org.apache.maven.plugins.shade.resource.ServicesResourceTransformer"/>
                    <transformer implementation="org.apache.maven.plugins.shade.resource.ManifestResourceTransformer">
                        <mainClass>com.example.Application</mainClass>
                    </transformer>
                </transformers>
            </configuration>
        </execution>
    </executions>
</plugin>
```

`maven-assembly-plugin` 是更通用的打包工具，除了内置的 `jar-with-dependencies` 等预设描述符（`descriptorRef`），还支持自定义描述符文件（`descriptor`），用于产出多种格式、自定义归档内部目录结构、或只打包部分文件：

```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-assembly-plugin</artifactId>
    <version>3.7.1</version>
    <configuration>
        <descriptors>
            <descriptor>src/assembly/assembly.xml</descriptor>
        </descriptors>
    </configuration>
    <executions>
        <execution>
            <phase>package</phase>
            <goals><goal>single</goal></goals>
        </execution>
    </executions>
</plugin>
```

```xml
<!-- src/assembly/assembly.xml -->
<assembly xmlns="http://maven.apache.org/ASSEMBLY/2.2.0">
    <id>dist</id>
    <formats>
        <format>zip</format>
    </formats>
    <includeBaseDirectory>false</includeBaseDirectory>
    <dependencySets>
        <dependencySet>
            <outputDirectory>lib</outputDirectory>
            <useProjectArtifact>true</useProjectArtifact>
        </dependencySet>
    </dependencySets>
    <fileSets>
        <fileSet>
            <directory>src/main/resources/config</directory>
            <outputDirectory>config</outputDirectory>
        </fileSet>
    </fileSets>
</assembly>
```

该描述符声明了：产出 `zip` 格式；依赖 jar 放入 `lib/` 目录；配置文件目录打入 `config/` 目录。相比内置 `descriptorRef`，自定义描述符能精确控制归档的目录结构，常用于需要产出特定分发包结构的场景。

::: warning
`maven-shade-plugin`/`maven-assembly-plugin` 与 `maven-dependency-plugin` 的 `copy-dependencies` 目标效果不同：前者将依赖解压后合并进单一归档（uber jar），可能因同名资源文件（如 `META-INF/services/*`）互相覆盖而丢失部分自动配置或服务注册；后者只是将依赖 jar 原样复制到外部目录，不涉及解压合并，也就没有这类冲突风险。选型时按是否需要"单文件分发"来取舍。
:::

### 资源打包补充：多组 `resources` 与 filtering

`maven-resources-plugin` 默认只处理 `src/main/resources` 下的全部文件，原样复制到 `target/classes`。若需要更精细的控制——例如部分文件需进行变量替换（`filtering`），部分文件不需要，或按构建参数动态选择需打入的文件——可在 `<build>` 下声明多组 `<resource>`，每组独立配置 `directory`/`includes`/`excludes`/`filtering`/`targetPath`：

```xml
<build>
    <resources>
        <!-- 第一组：静态前端资源，重定向到 static 目录，不做变量替换 -->
        <resource>
            <directory>src/main/webapp</directory>
            <targetPath>static</targetPath>
            <includes>
                <include>**/**</include>
            </includes>
            <excludes>
                <exclude>**/*.jsp</exclude>
            </excludes>
        </resource>
        <!-- 第二组：常规配置文件，开启filtering做变量替换，但排除需要单独处理的部分 -->
        <resource>
            <directory>src/main/resources</directory>
            <includes>
                <include>**/**</include>
            </includes>
            <excludes>
                <exclude>profiles/**/*.*</exclude>
                <exclude>templates/**</exclude>
            </excludes>
            <filtering>true</filtering>
        </resource>
        <!-- 第三组：模板文件本身含有与filtering语法冲突的占位符，单独关闭filtering -->
        <resource>
            <directory>src/main/resources</directory>
            <includes>
                <include>templates/**</include>
            </includes>
            <filtering>false</filtering>
        </resource>
        <!-- 第四组：按构建参数（如 -Denv=xxx 传入的属性）动态选择需打入哪个子目录下的文件 -->
        <resource>
            <directory>src/main/resources</directory>
            <includes>
                <include>profiles/${env}/*.xml</include>
            </includes>
            <filtering>true</filtering>
        </resource>
    </resources>
</build>
```

多组 `<resource>` 之间互不冲突，`includes`/`excludes` 各自独立生效；`filtering=true` 会使 Maven 将文件内容里 `${xxx}` 形式的占位符替换为对应的属性值，常用于让同一份源码在不同环境（开发/测试/生产）下产出内容不同的配置文件。

此外还有一种适用范围较窄但同样常见的场景：默认情况下，`src/main/java` 下的 XML 不会自动进入构建产物。若项目（如 MyBatis）将映射文件放在该目录，可通过一组 `<resource>` 显式纳入打包：

```xml
<build>
    <resources>
        <resource>
            <directory>src/main/java</directory>
            <includes>
                <include>**/*.xml</include>
            </includes>
        </resource>
    </resources>
</build>
```

## 十、工程实践：编写自定义 Maven 插件

标准插件覆盖不到的构建需求——例如构建期动态生成配置文件、按参数拉取额外资源——可通过编写自定义 Maven 插件解决。自定义插件本质上是一个实现特定接口的 Java 类（Mojo），由 Maven 在指定生命周期阶段调用执行。

### 编写步骤

1. 新建一个独立的 Maven 项目，`packaging` 设为 `maven-plugin`，引入 `maven-plugin-api` 和 `maven-plugin-annotations` 两个依赖
2. 定义一个继承 `AbstractMojo` 的类，用 `@Mojo(name = "目标名", defaultPhase = LifecyclePhase.PREPARE_PACKAGE)` 声明目标（goal）名称及其默认绑定的生命周期阶段
3. 需要从外部 pom 的 `<configuration>` 接收参数时，用 `@Parameter` 注解声明对应字段；具体处理逻辑写在 `execute()` 方法中

```java
@Mojo(name = "make-plugins", defaultPhase = LifecyclePhase.PREPARE_PACKAGE)
public class MakePluginsMojo extends AbstractMojo {

    @Parameter(property = "pid", required = true)
    private String pid;

    @Override
    public void execute() throws MojoExecutionException {
        // 具体处理逻辑：按 pid 参数生成/拉取所需资源，不在此展开
    }
}
```

### 使用方式

编写完成后通过 `mvn install` 发布到仓库（本地或私服），其他项目即可在 pom 中像使用任何标准插件一样引入并调用：

```xml
<plugin>
    <groupId>com.example</groupId>
    <artifactId>make-plugins-maven-plugin</artifactId>
    <version>1.0.0</version>
    <executions>
        <execution>
            <goals>
                <goal>make-plugins</goal>
            </goals>
            <configuration>
                <pid>${pid}</pid>
            </configuration>
        </execution>
    </executions>
</plugin>
```

### 适用场景与选型

企业级项目中常见此类自定义插件，例如按某个构建参数（如 `${pid}` 这类客户/环境标识）在构建期动态处理定制化资源。这类插件的目标名和内部逻辑因项目而异，把握上述骨架即可理解其基本机制，具体业务逻辑不必逐一展开。

::: tip
自定义插件与前文的 `maven-assembly-plugin` 自定义描述符、`resources` 多组配置本质上是同一思路的不同层级：前者是完全自行编写逻辑，后者是用标准插件的声明式配置满足需求。优先考虑声明式配置，仅当标准插件确实覆盖不到时才考虑自定义插件。
:::

## 十一、工程实践：Enforcer 构建约束

`maven-enforcer-plugin` 用于在构建阶段主动检查项目健康度，是企业级 Java 项目的常见配置。

### 常见规则示例

```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-enforcer-plugin</artifactId>
    <version>3.5.0</version>
    <executions>
        <execution>
            <id>enforce</id>
            <goals><goal>enforce</goal></goals>
            <configuration>
                <rules>
                    <!-- 依赖版本收敛：同一制品在依赖树中只允许出现一个版本 -->
                    <dependencyConvergence/>

                    <!-- 要求所有插件都显式声明版本，防止版本漂移 -->
                    <requirePluginVersions>
                        <banLatest>true</banLatest>
                        <banSnapshots>true</banSnapshots>
                    </requirePluginVersions>

                    <!-- 约束 JDK 版本范围 -->
                    <requireJavaVersion>
                        <version>[17,)</version>
                    </requireJavaVersion>

                    <!-- 约束 Maven 版本范围 -->
                    <requireMavenVersion>
                        <version>[3.8,)</version>
                    </requireMavenVersion>
                </rules>
            </configuration>
        </execution>
    </executions>
</plugin>
```

### 常用规则速查

| 规则 | 作用 |
|------|------|
| `dependencyConvergence` | 禁止依赖树中同一坐标出现多个版本 |
| `requirePluginVersions` | 要求所有插件显式声明版本 |
| `requireJavaVersion` | 约束构建所用 JDK 版本 |
| `requireMavenVersion` | 约束构建所用 Maven 版本 |
| `requireUpperBoundDeps` | 禁止传递依赖引入高于声明版本的制品 |
| `bannedDependencies` | 禁止特定依赖进入构建 |

::: warning
`dependencyConvergence` 在引入初期往往会暴露大量隐藏冲突，建议先在本地用 `mvn dependency:tree -Dverbose` 梳理，逐步通过 `dependencyManagement` 解决冲突，再在 CI 中开启强制校验。
:::

## 十三、工程实践：仓库与私服

Maven 仓库分两类：

- 本地仓库：缓存依赖与本地安装产物
- 远程仓库：Central、私服（Nexus/Artifactory）

Nexus 常见仓库类型：

| 类型 | 用途 |
|------|------|
| `proxy` | 代理外部远程仓库（如 Maven Central） |
| `group` | 聚合多个仓库，对客户端提供统一下载入口 |
| `hosted` | 托管组织内部制品（releases/snapshots） |

### 私服下载（settings.xml）

```xml
<mirrors>
    <mirror>
        <id>corp-nexus</id>
        <mirrorOf>central</mirrorOf>
        <url>https://nexus.example.com/repository/maven-public/</url>
    </mirror>
</mirrors>
```

### 私服部署（项目 pom.xml）

```xml
<distributionManagement>
    <repository>
        <id>corp-releases</id>
        <url>https://nexus.example.com/repository/maven-releases/</url>
    </repository>
    <snapshotRepository>
        <id>corp-snapshots</id>
        <url>https://nexus.example.com/repository/maven-snapshots/</url>
    </snapshotRepository>
</distributionManagement>
```

并在 `settings.xml` 配置对应 `server` 认证，`id` 必须一致。

### SNAPSHOT 与 Release 语义

Maven 用版本号后缀区分两种制品状态：

- **SNAPSHOT 版本**：版本号以 `-SNAPSHOT` 结尾，表示仍在开发中，允许覆盖发布。每次拉取时 Maven 会按一定策略检查远程仓库是否有更新。
- **Release 版本**：版本号不带 `-SNAPSHOT`（如 `1.2.3`），一旦发布即不可变，任何修改都必须升级版本号重新发布。

私服仓库通常也按这两类划分：

| 仓库类型 | 用途 |
|------|------|
| `maven-snapshots` | 接收 SNAPSHOT 版本，允许重复覆盖 |
| `maven-releases` | 接收 Release 版本，通常禁止覆盖 |

::: warning
不要把未稳定的 SNAPSHOT 版本带入生产发布；同时，Release 版本一旦发布就不要再修改同名制品，否则会导致依赖缓存不一致。
:::

## 十四、排错与诊断（高频问题）

### 依赖下载失败 / 一直失败不重试

可能原因：本地仓库残留 `.lastUpdated`。

处理思路：

1. 先确认坐标准确
2. 再确认网络和镜像
3. 最后清理目标依赖目录中的 `.lastUpdated` 文件

### 编译版本报错

检查：

- `mvn -v` 的 Java 版本
- `maven.compiler.release` 或 source/target 是否一致

### 类冲突 / NoSuchMethodError

大概率是依赖树冲突导致版本错配。

### 实用诊断命令

```bash
mvn help:effective-pom
mvn help:effective-settings
mvn help:active-profiles
mvn dependency:tree
mvn dependency:tree -Dverbose
mvn -X clean test
```

其中 `mvn dependency:tree -Dverbose` 适合定位"同坐标多版本竞争"场景，便于追溯冲突来源链路。

::: note 💡 实践提示
遇到疑难构建问题时，`effective-pom` 与 `dependency:tree` 往往是最优先的排查入口。
:::

## 十五、工程化基线清单

- 项目使用 Maven Wrapper（`mvnw`），锁定 Maven 版本
- 父工程统一依赖和插件版本
- 关键依赖走 BOM 或 `dependencyManagement`
- CI 中启用 `maven-enforcer-plugin`，约束 JDK/Maven 版本与依赖收敛
- 多环境差异通过 Profiles 管理，避免维护多套 POM
- 避免 `system` 作用域，优先上私服
- 不要把业务逻辑写进构建脚本
- CI 使用固定 Maven 与 JDK 版本
- 发布前至少执行 `./mvnw clean verify`
