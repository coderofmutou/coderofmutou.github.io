# Maven 依赖管理与项目构建

## 一、概念与定位：Maven 解决的问题

Maven 的核心价值在两件事：

1. **依赖管理**：自动下载、传递依赖、版本统一、冲突裁决
2. **构建管理**：统一编译、测试、打包、安装、部署流程

Maven 可视作“**项目构建与依赖的标准操作系统**”。

::: tip
在团队协作中，Maven 真正的价值不是“方便下载 jar”，而是“让每个人、每台机器、每条 CI 流水线执行相同构建动作并得到可复现结果”。
:::

## 二、环境准备与基础配置（Windows）

### 前置条件

- JDK 17+（建议长期支持版本）
- 正确设置 `JAVA_HOME`
- `mvn -v` 能输出版本信息

### 建议检查项

```bash
mvn -v
```

期望输出至少包含：

- Apache Maven 版本
- Java 版本
- 操作系统信息

### settings.xml 常见配置

通常在 Maven 安装目录下 `conf/settings.xml` 或用户目录下 `.m2/settings.xml` 中进行配置。

#### 本地仓库路径

```xml
<localRepository>D:/maven-repository</localRepository>
```

#### 镜像配置（示例）

```xml
<mirrors>
    <mirror>
        <id>aliyunmaven</id>
        <name>aliyun maven</name>
        <url>https://maven.aliyun.com/repository/public</url>
        <mirrorOf>central</mirrorOf>
    </mirror>
</mirrors>
```

::: warning
`settings.xml` 里的 `servers` 可能包含私服账号密码，不要提交到 Git 仓库。
:::

## 三、最小工程实践：从创建到运行

### 项目坐标（GAV）

Maven 项目由三个核心标识组成：

- `groupId`：组织/公司维度
- `artifactId`：模块名称
- `version`：版本号

完整坐标形式：`groupId:artifactId:version`

### GAV 命名约定（团队级）

在多人协作场景下，建议约定 GAV 命名规则，降低仓库污染与依赖歧义：

- `groupId`：建议使用反域名结构，如 `com.company.business`
- `artifactId`：建议使用“产品线-模块”语义，如 `order-service-api`
- `version`：建议遵循 `主版本.次版本.修订号`（如 `1.2.3`）

::: tip
命名规则的核心目标是“可检索、可区分、可演进”，而不是追求形式复杂度。
:::

### 最小可用 POM

```xml
<project xmlns="http://maven.apache.org/POM/4.0.0"
                 xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
                 xsi:schemaLocation="http://maven.apache.org/POM/4.0.0 https://maven.apache.org/xsd/maven-4.0.0.xsd">

        <modelVersion>4.0.0</modelVersion>

        <groupId>com.example</groupId>
        <artifactId>demo-maven</artifactId>
        <version>1.0.0-SNAPSHOT</version>

        <properties>
                <maven.compiler.release>17</maven.compiler.release>
                <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
                <project.reporting.outputEncoding>UTF-8</project.reporting.outputEncoding>
        </properties>

</project>
```

### 标准目录结构（强烈建议遵循）

```text
demo-maven/
├── pom.xml
└── src
        ├── main
        │   ├── java
        │   └── resources
        └── test
                ├── java
                └── resources
```

## 四、核心机制：生命周期与命令

Maven 内置三套生命周期：

- `clean`
- `default`
- `site`

日常开发最常用的是 `default` 生命周期中的这几步：

| 阶段 | 作用 |
|------|------|
| `compile` | 编译主代码 |
| `test` | 运行单元测试 |
| `package` | 打包（jar/war） |
| `verify` | 验证（含额外检查） |
| `install` | 安装到本地仓库 |
| `deploy` | 部署到远程仓库 |

### 常用命令组合

```bash
mvn clean
mvn clean test
mvn clean package
mvn clean install
mvn clean deploy
```

### 生命周期、命令、插件的关系

Maven 构建过程可以理解为三层映射：

1. 生命周期（Lifecycle）定义阶段序列
2. 命令（Phase）触发对应阶段及其前置阶段
3. 插件（Plugin Goal）最终完成实际动作

可简化为：`Lifecycle -> Phase -> Plugin Goal`。

例如执行 `mvn package` 时，本质上会触发 `package` 及其前置阶段，
并由编译、测试、打包相关插件完成具体工作。

::: tip
执行后置阶段会自动带上前置阶段。例如执行 `mvn package` 时，编译与测试相关步骤会先运行。
:::

## 五、核心机制：POM 关键结构块

### packaging

- `jar`：默认值，普通 Java 项目
- `war`：Web 项目
- `pom`：父工程/聚合工程

### dependencies 与 scope

```xml
<dependencies>
    <dependency>
        <groupId>org.junit.jupiter</groupId>
        <artifactId>junit-jupiter-api</artifactId>
        <version>5.11.0</version>
        <scope>test</scope>
    </dependency>
</dependencies>
```

常见作用域：

- `compile`：默认，编译/测试/运行都可见
- `test`：仅测试可见
- `provided`：编译与测试可见，运行由容器提供
- `runtime`：运行与测试可见，编译不可见

补充作用域（高频于工程治理）：

- `system`：与 `provided` 类似，但依赖来自本机路径；可移植性差，通常不建议使用
- `import`：仅用于在 `dependencyManagement` 中导入 BOM（`type=pom`）

### properties（统一变量）

```xml
<properties>
    <spring.version>6.1.10</spring.version>
    <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
</properties>
```

### build / plugins（构建行为定制）

```xml
<build>
    <plugins>
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-compiler-plugin</artifactId>
            <version>3.13.0</version>
            <configuration>
                <release>17</release>
            </configuration>
        </plugin>
    </plugins>
</build>
```

## 六、工程实践：依赖治理（传递、冲突、BOM）

### 传递依赖

如果 A 依赖 B，B 依赖 C，那么 A 默认会拿到 C（传递依赖）。

该机制提升了复用效率，但也可能导致依赖树快速膨胀。

### 冲突裁决规则

Maven 默认优先：

1. 路径更短的版本（nearest）
2. 路径同深度时，先声明优先

### 用 dependencyManagement 统一版本

```xml
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-dependencies</artifactId>
            <version>3.3.1</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>
```

子模块引用时可省略版本号。

### 排除不需要的传递依赖

```xml
<dependency>
    <groupId>com.example</groupId>
    <artifactId>demo-core</artifactId>
    <version>1.0.0</version>
    <exclusions>
        <exclusion>
            <groupId>commons-logging</groupId>
            <artifactId>commons-logging</artifactId>
        </exclusion>
    </exclusions>
</dependency>
```

::: warning
排除依赖前应先查看依赖树，避免运行期出现类缺失问题。
:::

### 传递终止补充：`optional`

当某依赖仅在当前模块内部需要、且不希望继续向下游模块传递时，可使用
`optional=true`：

```xml
<dependency>
    <groupId>com.alibaba</groupId>
    <artifactId>druid</artifactId>
    <version>1.2.15</version>
    <optional>true</optional>
</dependency>
```

该配置会阻断该依赖的默认传递，使用方需要时应显式声明。

## 七、工程实践：插件治理

推荐在父工程通过 `pluginManagement` 锁定插件版本：

```xml
<build>
    <pluginManagement>
        <plugins>
            <plugin>
                <groupId>org.apache.maven.plugins</groupId>
                <artifactId>maven-compiler-plugin</artifactId>
                <version>3.13.0</version>
            </plugin>
            <plugin>
                <groupId>org.apache.maven.plugins</groupId>
                <artifactId>maven-surefire-plugin</artifactId>
                <version>3.2.5</version>
            </plugin>
        </plugins>
    </pluginManagement>
</build>
```

这样可以避免不同机器默认插件版本不一致。

### 常用插件速查

| 插件 | 作用 | 典型配置项 |
|------|------|-----------|
| `maven-compiler-plugin` | 设置编译 source/target/release | `<release>17</release>` |
| `maven-surefire-plugin` | 运行单元测试 | 指定包含/排除测试类 |
| `maven-failsafe-plugin` | 运行集成测试 | 绑定 `integration-test` 阶段 |
| `maven-source-plugin` | 打包源码 | 绑定 `package` 阶段生成 `-sources.jar` |
| `maven-javadoc-plugin` | 打包 API 文档 | 生成 `-javadoc.jar` |
| `maven-checkstyle-plugin` | 代码风格检查 | 指定 checkstyle 规则文件 |
| `spotbugs-maven-plugin` | 静态缺陷扫描 | 绑定 `verify` 阶段 |

示例：同时生成源码包与文档包。

```xml
<build>
    <plugins>
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-source-plugin</artifactId>
            <version>3.3.1</version>
            <executions>
                <execution>
                    <id>attach-sources</id>
                    <goals>
                        <goal>jar-no-fork</goal>
                    </goals>
                </execution>
            </executions>
        </plugin>
    </plugins>
</build>
```

::: tip
插件版本建议在父工程通过 `pluginManagement` 统一锁定，而不是在每个子模块重复写版本号。
:::

### 资源打包补充：`src/main/java` 下 XML 文件

默认情况下，`src/main/java` 下的 XML 不会自动进入构建产物。
若项目（如 MyBatis）将映射文件放在该目录，可通过 `resources` 显式纳入打包：

```xml
<build>
    <resources>
        <resource>
            <directory>src/main/java</directory>
            <includes>
                <include>**/*.xml</include>
            </includes>
        </resource>
    </resources>
</build>
```

## 八、工程实践：多模块组织（继承 + 聚合）

### 父工程（聚合 + 统一管理）

```xml
<project>
    <modelVersion>4.0.0</modelVersion>
    <groupId>com.example</groupId>
    <artifactId>demo-parent</artifactId>
    <version>1.0.0-SNAPSHOT</version>
    <packaging>pom</packaging>

    <modules>
        <module>common-service</module>
        <module>user-service</module>
        <module>order-service</module>
    </modules>
</project>
```

### 子工程（继承父工程）

```xml
<project>
    <modelVersion>4.0.0</modelVersion>

    <parent>
        <groupId>com.example</groupId>
        <artifactId>demo-parent</artifactId>
        <version>1.0.0-SNAPSHOT</version>
    </parent>

    <artifactId>user-service</artifactId>
</project>
```

### 一条命令构建全模块

```bash
mvn clean verify
```

## 九、工程实践：仓库与私服

Maven 仓库分两类：

- 本地仓库：缓存依赖与本地安装产物
- 远程仓库：Central、私服（Nexus/Artifactory）

Nexus 常见仓库类型：

| 类型 | 用途 |
|------|------|
| `proxy` | 代理外部远程仓库（如 Maven Central） |
| `group` | 聚合多个仓库，对客户端提供统一下载入口 |
| `hosted` | 托管组织内部制品（releases/snapshots） |

### 私服下载（settings.xml）

```xml
<mirrors>
    <mirror>
        <id>corp-nexus</id>
        <mirrorOf>central</mirrorOf>
        <url>https://nexus.example.com/repository/maven-public/</url>
    </mirror>
</mirrors>
```

### 私服部署（项目 pom.xml）

```xml
<distributionManagement>
    <repository>
        <id>corp-releases</id>
        <url>https://nexus.example.com/repository/maven-releases/</url>
    </repository>
    <snapshotRepository>
        <id>corp-snapshots</id>
        <url>https://nexus.example.com/repository/maven-snapshots/</url>
    </snapshotRepository>
</distributionManagement>
```

并在 `settings.xml` 配置对应 `server` 认证，`id` 必须一致。

### SNAPSHOT 与 Release 语义

Maven 用版本号后缀区分两种制品状态：

- **SNAPSHOT 版本**：版本号以 `-SNAPSHOT` 结尾，表示仍在开发中，允许覆盖发布。每次拉取时 Maven 会按一定策略检查远程仓库是否有更新。
- **Release 版本**：版本号不带 `-SNAPSHOT`（如 `1.2.3`），一旦发布即不可变，任何修改都必须升级版本号重新发布。

私服仓库通常也按这两类划分：

| 仓库类型 | 用途 |
|------|------|
| `maven-snapshots` | 接收 SNAPSHOT 版本，允许重复覆盖 |
| `maven-releases` | 接收 Release 版本，通常禁止覆盖 |

::: warning
不要把未稳定的 SNAPSHOT 版本带入生产发布；同时，Release 版本一旦发布就不要再修改同名制品，否则会导致依赖缓存不一致。
:::

## 十、排错与诊断（高频问题）

### 依赖下载失败 / 一直失败不重试

可能原因：本地仓库残留 `.lastUpdated`。

处理思路：

1. 先确认坐标准确
2. 再确认网络和镜像
3. 最后清理目标依赖目录中的 `.lastUpdated` 文件

### 编译版本报错

检查：

- `mvn -v` 的 Java 版本
- `maven.compiler.release` 或 source/target 是否一致

### 类冲突 / NoSuchMethodError

大概率是依赖树冲突导致版本错配。

### 实用诊断命令

```bash
mvn help:effective-pom
mvn help:effective-settings
mvn dependency:tree
mvn dependency:tree -Dverbose
mvn -X clean test
```

其中 `mvn dependency:tree -Dverbose` 适合定位“同坐标多版本竞争”场景，便于追溯冲突来源链路。

::: note 💡 实践提示
遇到疑难构建问题时，`effective-pom` 与 `dependency:tree` 往往是最优先的排查入口。
:::

## 十一、工程化基线清单

- 父工程统一依赖和插件版本
- 关键依赖走 BOM 或 `dependencyManagement`
- 避免 `system` 作用域，优先上私服
- 不要把业务逻辑写进构建脚本
- CI 使用固定 Maven 与 JDK 版本
- 发布前至少执行 `mvn clean verify`
