# Markdown 写作规范

适用范围：`docs/` 下所有 Markdown 文档。

---

## 一、目标

写作时同时满足这 4 个目标：

1. **可读**：面向人阅读，结构清晰，不堆原文。
2. **可检索**：标题、术语、分类、标签便于后续搜索。
3. **可维护**：后续补充版本差异、拆分章节时不易失控。
4. **可发布**：兼容 VuePress / Vdoing 渲染与现有自动化检查。

---

## 二、内容组织原则

### 1. 优先讲清“是什么、为什么、怎么用、注意什么”

每个知识点尽量按以下顺序组织：

1. 是什么
2. 为什么需要它
3. 怎么使用
4. 注意事项 / 常见坑 / 最佳实践

适合知识型笔记，不建议大量堆砌课程口播稿、广告语、寒暄语。

### 2. 先给结论，再给细节

对于概念、选型、排错结论，优先给一句总结，再展开细节。

```markdown
## 事务传播行为

一句话理解：事务传播行为决定“当前方法如何加入已有事务，或是否新开事务”。

### REQUIRED
...
```

### 3. 代码优先于大段复述

- 有最小可运行示例时，优先保留示例。
- 与代码一一对应的解释保留。
- 纯重复定义、口语化铺垫、营销话术删除。

### 4. 版本信息必须显式标注

当内容受版本影响时，必须写清版本边界，例如：

- `Spring Security 5.x`
- `Spring Security 6.x`
- `JDK 17+`
- `VuePress 1.x`

不要把不同版本语义混写在同一段中，除非你明确标注差异。

## 三、标题与结构规则

### 1. 一个文件只能有一个 H1

- `frontmatter.title` 与 H1 保持一致。
- 正文从 H2 开始。

```markdown
# Spring Security

## 核心概念

### 认证
```

### 2. 标题要“可检索”

标题应包含实际术语，不要只写“补充”“总结”“问题”。

```markdown
# ❌ 不推荐
## 注意事项

# ✅ 推荐
## Bean 生命周期中的循环依赖问题
```

### 3. 同级标题粒度保持一致

同一层级下不要混用“概念名”“完整句子”“碎片提示”。

### 4. 文件与目录命名优先遵循仓库约定

通用 Markdown 规则常会建议使用 `主题-子主题.md` 这类主题优先命名，
但本仓库是 **结构化知识库**，文件命名优先遵循现有目录与排序规则：

- 目录与文件使用 `{两位数字}.{中文标题}` 形式控制排序
- 一级、二级目录与 `docs/00.目录页/`、`docs/.vuepress/config.ts` 保持一致
- 已发布文章若已绑定稳定 `permalink`，不要随意改名

结论：

- **仓库内文章**：优先遵循本仓库编号命名体系
- **通用 Markdown 命名建议**：仅作为无仓库约束时的默认参考

## 四、段落与表达规范

### 1. 一段只讲一件事

- 长段落尽量拆分。
- 列点优先于“逗号接力赛”。

### 2. 避免课程腔与口语噪声

应删除或改写：

- “大家可以看到……”
- “老师这里演示一下……”
- “这个非常非常重要哈……”
- “懂的都懂”

### 3. 个人理解要与原始知识分层

个人补充建议使用引用块或专门小节，避免与原始知识混在一起。

```markdown
> 💡 个人注：这里的核心不是记住 API，而是理解过滤器链的执行顺序。
```

### 4. 结论、风险、建议要有视觉层次

- 普通说明：普通段落
- 提示性信息：容器或引用块
- 风险与陷阱：`warning` / `danger` 容器
- 最佳实践：`tip` / `note` 容器

### 5. 引用来源要可追溯、可公开访问

如果内容来自官方文档、书籍、课程或第三方文章，应写明出处；
但**不要引用个人电脑上的绝对路径文件**。

推荐写法：

```markdown
> 引自 [VuePress Markdown 扩展文档](https://v1.vuepress.vuejs.org/guide/markdown.html)
```

或：

```markdown
参考资料：

- [Spring Security 官方文档](https://spring.io/projects/spring-security)
- [VuePress 文档](https://v1.vuepress.vuejs.org/)
```

不推荐写法：

```markdown
参考来源：`D:\xx\xx\xxx.md`
```

## 五、行内元素使用规则

### 1. 行内代码只用于“技术实体”

以下内容建议使用反引号：

- 代码
- 命令
- 类名 / 方法名 / 变量名
- 配置项
- 文件路径
- URL 路径片段 / permalink

```markdown
使用 `@Transactional` 开启事务。
运行 `npm run build` 做最终验收。
修改 `docs/.vuepress/config.ts`。
```

普通英文与数字不必滥用反引号：

```markdown
我们使用 Java 17 与 Spring Boot 3。
```

### 2. 强调要克制

- 优先使用 `**加粗**` 标记关键信息。
- 需要斜体时统一使用 `*斜体*` 或 `_斜体_` 其中一种，不要随手混搭。
- 不要连续多句全加粗。
- 不要把整段都写成强调。

### 3. 链接文本要有语义

```markdown
# ❌ 不推荐
[点击这里](https://example.com)

# ✅ 推荐
[VuePress Markdown 扩展文档](https://v1.vuepress.vuejs.org/guide/markdown.html)
```

优先使用**内联超链接**，避免把裸 URL 大段堆在正文里。

## 六、列表、表格、代码块

### 1. 列表适合表达并列关系

- 列表项尽量句式一致。
- 一个列表里不要混入完全不同粒度的内容。

### 1.1 段落与空行要稳定

- 段落之间空一行。
- 标题上下保留空行。
- 列表、表格、代码块与正文之间保留合理空行，避免内容粘连。

### 2. 表格只在“字段对比、参数汇总、能力矩阵”时使用

如果内容较长、需要解释较多，优先用二级标题加列表，不要强塞表格。

### 3. 代码块必须带语言标识

````markdown
```java
public class UserService {
}
```
````

当前仓库已对部分语言别名做兼容映射，但写作时仍建议优先使用标准语言名：

- `java`
- `xml`
- `yaml`
- `bash`
- `sql`
- `javascript`
- `typescript`

### 4. 示例代码尽量最小化

- 只保留和当前知识点直接相关的部分。
- 大段样板代码优先裁剪。
- 必要时补一行“省略无关代码”。

### 5. 行宽与换行

- 单行尽量控制在约 80–100 个字符内。
- 超长句、超长列表项、超长说明尽量手动换行。
- 换行的目标是提高可读性与 diff 可维护性，不是机械断句。

## 七、图片与图示内容

### 1. 图片不是越多越好

优先保留：

- 架构图
- 流程图
- 结果对比图
- 难以纯文字表达的界面截图

优先删除：

- 重复截图
- 低信息密度截图
- 广告、课程封面、讲师介绍图

### 2. 图片下方最好补一句说明

说明图片在证明什么，不要只贴图不解释。

### 3. 路径与格式遵循仓库约定

- 图片放同级 `.assets/` 目录。
- 优先使用本地图片与 `.webp`。
- 使用相对路径，不要写绝对路径。

---

## 八、知识库写作禁忌

以下内容应尽量避免：

1. 直接粘贴未经整理的课程逐字稿
2. 无版本前提的结论
3. 标题党式命名
4. 多层嵌套但没有主线的目录结构
5. 大量外链图片未本地化
6. 只有代码没有解释，或只有解释没有示例
7. 长篇堆砌 without summary（是的，这种中英混搭正是我们要避免的）
8. 用本机绝对路径、私有截图路径、临时目录路径做“参考来源”

## 十、写作自检清单

- [ ] 标题是否能体现知识点，而不是“总结 / 补充 / 说明”
- [ ] 是否先给结论，再展开解释
- [ ] 是否删除了课程口语、广告语、重复描述
- [ ] 是否写清版本边界
- [ ] 是否提供了必要示例，而不是只有抽象定义
- [ ] 行内代码、代码块、图片、链接是否都各司其职
- [ ] 是否将个人理解与客观知识做了分层
- [ ] 引用来源是否可追溯且不依赖本机绝对路径
- [ ] 文件命名与目录位置是否符合当前仓库的编号与导航规则
- [ ] 是否适合在 VuePress / Vdoing 中长期维护
