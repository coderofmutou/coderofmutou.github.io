# Markdown 写作规范

适用范围：`docs/` 下所有 Markdown 文档。

---

## 一、目标

写作时同时满足这 4 个目标：

1. **可读**：面向人阅读，结构清晰，不堆原文。
2. **可检索**：标题、术语、分类、标签便于后续搜索。
3. **可维护**：后续补充版本差异、拆分章节时不易失控。
4. **可发布**：兼容 VuePress / Vdoing 渲染与现有自动化检查。

---

## 二、内容组织原则

### 1. 优先讲清“是什么、为什么、怎么用、注意什么”

每个知识点尽量按以下顺序组织：

1. 是什么
2. 为什么需要它
3. 怎么使用
4. 注意事项 / 常见坑 / 最佳实践

适合知识型笔记，不建议大量堆砌课程口播稿、广告语、寒暄语。

### 2. 先给结论，再给细节

对于概念、选型、排错结论，优先给一句总结，再展开细节。

```markdown
## 事务传播行为

一句话理解：事务传播行为决定“当前方法如何加入已有事务，或是否新开事务”。

### REQUIRED
...
```

### 3. 代码优先于大段复述

- 有最小可运行示例时，优先保留示例。
- 与代码一一对应的解释保留。
- 纯重复定义、口语化铺垫、营销话术删除。

### 4. 版本信息必须显式标注

当内容受版本影响时，必须写清版本边界，例如：

- `Spring Security 5.x`
- `Spring Security 6.x`
- `JDK 17+`
- `VuePress 1.x`

不要把不同版本语义混写在同一段中，除非你明确标注差异。

## 三、标题与结构规则

### 1. 一个文件只能有一个 H1

- `frontmatter.title` 与 H1 保持一致。
- 正文从 H2 开始。

```markdown
# Spring Security

## 核心概念

### 认证
```

### 2. 标题要“可检索”

标题应包含实际术语，不要只写“补充”“总结”“问题”。

```markdown
# ❌ 不推荐
## 注意事项

# ✅ 推荐
## Bean 生命周期中的循环依赖问题
```

### 3. 同级标题粒度保持一致

同一层级下不要混用“概念名”“完整句子”“碎片提示”。

### 4. 目录层级过深时，改用 HTML 标签控制 TOC 深度

VuePress 只从 Markdown `#` 语法提取标题，`<h5>` 等 HTML 标签**渲染效果相同但不进入 TOC**。因此当文档层级较深、继续加 `#` 级会让侧边栏条目冗长失焦时，可把不需要出现在导航中的子级标题改用 HTML 标签：

```markdown
<!-- ❌ Markdown 标题，会被计入 TOC -->
##### 3.2.2 Java 代码

<!-- ✅ HTML 标签，保持视觉层次，不进入 TOC -->
<h5>3.2.2 Java 代码</h5>
```

**判断标准**：

- 换的触发点不是"到第 5 级"，而是**"目录条目已够用，再往下只会让导航失焦"**
- 该标题用于**组织内部内容**（代码示例、步骤对比）而非导航锚点时，优先用 HTML 标签
- 层级超过 4 级先考虑**拆分文件**；确属同一篇笔记再用 HTML 标签

**注意**：HTML 标题无自动锚点，如需外链，手动加 `id`：`<h5 id="xxx">标题</h5>`。

### 5. 文件与目录命名优先遵循仓库约定

通用 Markdown 规则常会建议使用 `主题-子主题.md` 这类主题优先命名，
但本仓库是 **结构化知识库**，文件命名优先遵循现有目录与排序规则：

- 目录与文件使用 `{两位数字}.{中文标题}` 形式控制排序
- 一级、二级目录与 `docs/00.目录页/`、`docs/.vuepress/config.ts` 保持一致
- 已发布文章若已绑定稳定 `permalink`，不要随意改名

结论：

- **仓库内文章**：优先遵循本仓库编号命名体系
- **通用 Markdown 命名建议**：仅作为无仓库约束时的默认参考

## 四、段落与表达规范

### 1. 一段只讲一件事

- 长段落尽量拆分。
- 列点优先于“逗号接力赛”。

### 2. 避免课程腔与口语噪声

应删除或改写：

- “大家可以看到……”
- “老师这里演示一下……”
- “这个非常非常重要哈……”
- “懂的都懂”

### 3. 个人理解要与原始知识分层

个人补充建议使用 `::: note` 容器或专门小节，避免与原始知识混在一起。

```markdown
::: note 💡 个人注
这里的核心不是记住 API，而是理解过滤器链的执行顺序。
:::
```

### 4. 结论、风险、建议要有视觉层次

- 普通说明：普通段落
- 提示性信息、最佳实践、个人总结：`tip` / `note` 容器
- 风险与陷阱：`warning` / `danger` 容器
- 外部引用：`>` 引用块（须注明来源）

### 5. 引用来源要可追溯、可公开访问

如果内容来自官方文档、书籍、课程或第三方文章，应写明出处；
但**不要引用个人电脑上的绝对路径文件**。

推荐写法：

```markdown
> 引自 [VuePress Markdown 扩展文档](https://v1.vuepress.vuejs.org/guide/markdown.html)
```

或：

```markdown
参考资料：

- [Spring Security 官方文档](https://spring.io/projects/spring-security)
- [VuePress 文档](https://v1.vuepress.vuejs.org/)
```

不推荐写法：

```markdown
参考来源：`D:\xx\xx\xxx.md`
```

## 五、行内元素使用规则

### 1. 行内代码只用于“技术实体”

以下内容建议使用反引号：

- 代码
- 命令
- 类名 / 方法名 / 变量名
- 配置项
- 文件路径
- URL 路径片段 / permalink

```markdown
使用 `@Transactional` 开启事务。
运行 `npm run build` 做最终验收。
修改 `docs/.vuepress/config.ts`。
```

普通英文与数字不必滥用反引号：

```markdown
我们使用 Java 17 与 Spring Boot 3。
```

### 2. 强调要克制

- 优先使用 `**加粗**` 标记关键信息。
- 需要斜体时统一使用 `*斜体*` 或 `_斜体_` 其中一种，不要随手混搭。
- 不要连续多句全加粗。
- 不要把整段都写成强调。

### 3. 链接文本要有语义

```markdown
# ❌ 不推荐
[点击这里](https://example.com)

# ✅ 推荐
[VuePress Markdown 扩展文档](https://v1.vuepress.vuejs.org/guide/markdown.html)
```

优先使用**内联超链接**，避免把裸 URL 大段堆在正文里。

## 六、列表、表格、代码块

### 1. 列表适合表达并列关系

- 列表项尽量句式一致。
- 一个列表里不要混入完全不同粒度的内容。

### 2. 段落与空行要稳定

- 段落之间空一行。
- 标题上下保留空行。
- 列表、表格、代码块与正文之间保留合理空行，避免内容粘连。

### 3. 表格只在”字段对比、参数汇总、能力矩阵”时使用

如果内容较长、需要解释较多，优先用二级标题加列表，不要强塞表格。

### 4. 代码块必须带语言标识

````markdown
```java
public class UserService {
}
```
````

当前仓库已对部分语言别名做兼容映射，但写作时仍建议优先使用标准语言名：

- `java`
- `xml`
- `yaml`
- `bash`
- `sql`
- `javascript`
- `typescript`

### 5. 示例代码尽量最小化

- 只保留和当前知识点直接相关的部分。
- 大段样板代码优先裁剪。
- 必要时补一行“省略无关代码”。

### 6. 行宽与换行

- 单行尽量控制在约 80–100 个字符内。
- 超长句、超长列表项、超长说明尽量手动换行。
- 换行的目标是提高可读性与 diff 可维护性，不是机械断句。

---

## 七、图片与图示内容

### 1. 图片不是越多越好

优先保留：

- 架构图
- 流程图
- 结果对比图
- 难以纯文字表达的界面截图

优先删除：

- 重复截图
- 低信息密度截图
- 广告、课程封面、讲师介绍图

### 2. 图片下方最好补一句说明

说明图片在证明什么，不要只贴图不解释。

### 3. 路径与格式遵循仓库约定

- 图片放同级 `.assets/` 目录。
- 优先使用本地图片与 `.webp`。
- 使用相对路径，不要写绝对路径。

---

## 八、知识库写作禁忌

以下内容应尽量避免：

1. 直接粘贴未经整理的课程逐字稿
2. 无版本前提的结论
3. 标题党式命名
4. 多层嵌套但没有主线的目录结构
5. 大量外链图片未本地化
6. 只有代码没有解释，或只有解释没有示例
7. 长篇堆砌 without summary（是的，这种中英混搭正是我们要避免的）
8. 用本机绝对路径、私有截图路径、临时目录路径做“参考来源”

## 十、内容优化检查项

对已有笔记做内容优化时，按以下分类执行。优化目标是提升可读性和长期可维护性：**主动删除无效噪声和真正重复的内容，保留有价值的详细说明与代码示例**。判断标准是"这段内容对读者有价值吗"，而不是"文档够不够短"。

### agent 可自动执行

| 类型 | 判断标准 | 处理方式 |
|------|---------|---------|
| 无关噪声 | 课程广告、讲师介绍、"大家好我是XX"、"本节课我们学习"、"非常重要哈"等口语铺垫 | 直接删除 |
| 无效引用块 | 整段代码被套进 `>` 引用块；或普通正文（非提示/引用）误用引用块格式 | 去掉 `>` 前缀，恢复为普通段落或代码块 |
| 个人注/提示未容器化 | `>` 引用块内容为个人总结、经验、建议、注意事项，而非外部引用（无来源链接、语气为第一人称或主观分析） | 根据内容语义转换为对应容器（建议→`tip`，总结→`note`，注意→`warning`） |
| 空洞总结段 | "本节我们学习了XXX，掌握了XXX，希望大家..."纯情感收尾；**含具体内容（列表/代码/要点）的结尾不动** | 删除纯情感收尾段 |
| 代码块缺语言标注 | ` ``` ` 后无 `java`/`xml`/`yaml`/`bash`/`sql` 等标识 | 根据上下文启发式推断语言类型，自动补全 |
| 标题层级跳跃 | H1 直接跳 H3/H4、同级标题层次不一致 | 补齐中间层级，保持 H1→H2→H3 渐进结构 |
| 链接文本无语义 | 链接文字为 `点击这里`、`链接`、`这里`、`here` 等无描述性内容 | 将链接文本替换为描述链接目标的语义文字 |

### agent 评估后决策

以下类型不再一律等待确认——agent 先自评改动价值，再选择行动方式：

- **直接执行**：改动目标客观明确、收益显著、不依赖用户偏好 → 直接修改原文，在文件摘要中注明
- **列出候选**：涉及主观取舍、大幅内容增删或重组、agent 自身不确定 → 列入该文件的改动摘要，附建议，由用户决定

| 类型 | 判断标准 | 执行方式 |
|------|---------|---------|
| 截图评估 | 图片内容为代码输出、配置项列表、纯文字结果（可用代码块替代），或低信息密度界面截图 | 始终列出候选：扫描所有截图，附可替换性建议，由用户逐项确认（涉及删除资产，不直接执行） |
| 冗余表述 | 同一概念在同一篇文章中被完整解释了两次以上 | 两版重叠度高（≥80%）且一版明显更清晰 → 直接合并，保留最佳版本；两版各有侧重 → 列出候选，注明重叠程度 |
| 代码与说明脱节 | 代码块前后无任何说明段落，读者无法理解该代码解决什么问题 | 始终列出候选，提示补充使用场景（不直接添加，无法确定代码意图） |
| 版本未标注 | 含版本号关键词（如 `Spring Boot 3`、`JDK 17`）但缺少版本适用范围说明 | 始终列出候选，提示补充版本边界 |
| 外链失效 | HTTP 链接返回 404 或无法访问 | 始终列出候选，用户决定删除或替换 |
| 技术名词格式 | 常见技术名词大小写/格式有误，如 `SpringBoot`→`Spring Boot`、`Github`→`GitHub`、`Mybatis`→`MyBatis`、`javascript`→`JavaScript` | 有明确标准写法 → 直接替换；拿不准的 → 列出候选 |
| 过长段落 | 单个段落超过约 300 字且无换行，可读性差 | 段落内含多个独立陈述、断句位置明显 → 直接拆分；段落是连贯论证 → 列出候选 |

**判断示例（灰色地带参考）：**

直接执行的典型情形：
- § 3.2 和 § 3.5 都包含对 `@Transactional` 传播行为的完整定义 + 相同代码示例，措辞高度重复 → 直接合并，保留表达更清晰的一版
- 文中出现 `Github`、`Mybatis`、`SpringBoot` → 有明确标准写法，直接替换为 `GitHub` / `MyBatis` / `Spring Boot`
- 某节有一段 400 字描述"概念定义、使用场景、注意事项"三件独立的事，各有自然断点 → 直接拆分为三段

列出候选的典型情形：
- § 2.1 简短提到事务隔离级别的概念，§ 4.3 深入展开各级别的区别 → 两版层层递进，非真正重复，列候选注明"渐进铺垫"
- 某段 350 字是连续的因果推导，整段连读才能理解 → 列候选，由用户决定是否拆分
- 某段内容表面重复前面的概念，但角度不同或加入了新代码示例 → 各有侧重，列候选而非直接合并

> 处理过程中不中断询问，每完成一个文件只输出一行进度标记（如 `✓ 文件名 已处理`）。所有文件的内容写作全部完成后（进入 Step 2 之前），统一输出两部分：

① **各文件改动摘要**：每个文件执行了哪些修改（可回溯）

② **汇总候选项清单**：全部文件的候选项按类型分组，每项标注所在文件与位置，供用户统一决策是否继续处理

## 十一、写作自检清单

- [ ] 标题是否能体现知识点，而不是“总结 / 补充 / 说明”
- [ ] 是否先给结论，再展开解释
- [ ] 是否删除了课程口语、广告语、重复描述
- [ ] 是否写清版本边界
- [ ] 是否提供了必要示例，而不是只有抽象定义
- [ ] 行内代码、代码块、图片、链接是否都各司其职
- [ ] 是否将个人理解与客观知识做了分层
- [ ] 引用来源是否可追溯且不依赖本机绝对路径
- [ ] 文件命名与目录位置是否符合当前仓库的编号与导航规则
- [ ] 是否适合在 VuePress / Vdoing 中长期维护
