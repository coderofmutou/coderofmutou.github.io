# Git Commit Message 规范

在已有中文短句风格基础上，引入 `[类型]` 前缀分类，便于快速扫描 git log，同时保持中文可读性。

---

## 格式

```
[类型] {操作动词} {操作对象}[，{补充说明}]
```

- `[类型]` 为固定分类前缀，见下方类型表
- 操作描述全部使用**中文**
- 主体内容控制在 **50 字以内**，超出时用逗号分隔补充说明
- 单次提交只做一件事，多件事分多次提交

---

## 类型前缀

| 前缀 | 适用场景 |
|------|----------|
| `[笔记]` | 新增、更新、重构笔记内容 |
| `[图片]` | 图片本地化、转 webp、清理未使用图片 |
| `[配置]` | 修改 config.ts、导航菜单、frontmatter |
| `[脚本]` | 修改 utils/ 下的工具脚本 |
| `[依赖]` | 升级/更换 npm 包或运行环境 |
| `[修复]` | 修正笔记错误内容、修复脚本 bug、修复构建问题 |
| `[技能]` | 更新 .claude/skills/ 下的技能文件 |

## 各工作流对应的 commit 示例

### 新增笔记（完整新文档）

```
[笔记] 新增 {技术名称} 笔记
```

### 系列笔记（分多次提交，每次必须注明章节）

```
[笔记] 新增 {系列名称} 笔记（{章节描述}）

示例：
[笔记] 新增 JavaWeb 技术笔记（第01-03章）
[笔记] 新增 JavaWeb 技术笔记（第07章_Servlet）
```

> ⚠️ 系列笔记**禁止**多次使用完全相同的 message（如连续10次 `新增 JavaWeb 技术笔记`），必须区分章节范围。

### 整合他人笔记

```
[笔记] 新增 {技术名称} 笔记，整合自 {来源}

示例：
[笔记] 新增 Redis 笔记，整合自尚硅谷教程
```

### 新增版本适配内容

```
[笔记] 更新 {技术名称} 笔记，新增 {版本} 章节

示例：
[笔记] 更新 Spring Security 笔记，新增 6.x 版本章节
[笔记] 更新 MyBatis 笔记，新增 Spring Boot 3.x 集成方式
```

### 图片处理（可合并为一次提交）

```
[图片] 优化 {技术名称} 笔记图片，本地化外链并转换为 webp
```

### 更新已有笔记

```
更新 {技术名称} 笔记，{具体变更}

示例：
更新 SpringCloud 笔记，补充 Gateway 章节
更新 MyBatis-Plus 笔记，同步至 3.5.x 版本变更
```

### config.ts 导航变更

```
更新导航菜单，{具体变更}

示例：
更新导航菜单，新增 Redis 子项
更新导航菜单，调整微服务生态分类结构
```

## 反面示例（避免）

| 不好的写法 | 问题 | 改为 |
|-----------|------|------|
| `update` | 英文且无说明 | `[笔记] 更新 MyBatis 笔记` |
| `笔记` | 过于简短 | `[笔记] 新增 Redis 笔记` |
| `格式调整` | 没说调整了什么 | `[笔记] 调整 JDBC 笔记标题层级和代码块格式` |
| `修改了很多东西` | 描述模糊 | 拆成多次独立提交 |
| 连续10次 `新增 JavaWeb 技术笔记`（无章节区分）| 无法区分内容范围 | `[笔记] 新增 JavaWeb 技术笔记（第07章_Servlet）` |
| `wip` / `临时提交` | 提交了未完成的工作 | 完成后再提交，或用 `git stash` 暂存 |

---

## 提交时机建议

- **新增笔记**：完成一个完整章节后提交，不要每写几段就提交一次
- **图片处理**：处理完一篇文档的全部图片后统一提交
- **config.ts 改动**：与对应的笔记文件一起提交（同一次操作的变更在同一个 commit）
- **工具脚本修改**：单独提交，不要和笔记内容混在一起
