Gradle 依赖管理与项目构建

# Gradle 依赖管理与项目构建

# 一、概念与定位：Gradle 解决的问题

Gradle 的核心价值同样落在两件事：

  1. 依赖管理：声明依赖、统一版本、冲突治理
  2. 构建管理：任务编排、测试执行、打包发布、流水线集成

相比 Maven，Gradle 的优势在于任务模型与 DSL 可编排能力； 相比脚本化构建，Gradle 的优势在于标准化与可维护性。

常见使用场景：

  • 多模块项目统一构建
  • 依赖版本治理与约束
  • 编译、测试、打包、发布自动化
  • CI/CD 流水线的标准任务入口

提示

在团队协作中，Gradle 的价值不只是"构建更快"，更是"构建行为可复用、可审计、可演进"。

# 二、环境准备与基础配置（Windows）

# 前置条件

  • JDK 17+
  • JAVA_HOME 正确配置

# 建议检查项

gradle -v
1

期望输出包含：

  • Gradle 版本
  • JVM 版本
  • 操作系统信息

# Wrapper 优先原则

建议优先使用 Gradle Wrapper（gradlew）而不是系统全局 Gradle， 避免团队机器与 CI 的 Gradle 版本不一致。

# 需要本机已安装 Gradle
gradle wrapper --gradle-version 8.12
1
2

建议将以下 Wrapper 文件纳入版本控制：

  • gradlew
  • gradlew.bat
  • gradle/wrapper/gradle-wrapper.properties
  • gradle/wrapper/gradle-wrapper.jar

# 三、gradle.properties 常用配置

gradle.properties 是 Gradle 的键值对配置文件，影响构建性能与行为。 可放置在项目根目录（项目级）或 ~/.gradle/ 目录下（用户级）。

# 开启 Gradle Daemon，避免每次构建重启 JVM（Gradle 3.0+ 默认已启用）
org.gradle.daemon=true

# 并行构建：多模块项目可并行执行独立子项目的任务
org.gradle.parallel=true

# 构建缓存：复用历史任务输出，跳过不必要的重复执行
org.gradle.caching=true

# 配置缓存（Gradle 8.1+ 稳定）：跳过构建配置阶段，直接加载任务图
org.gradle.configuration-cache=true

# JVM 堆大小，大型项目可适当调高
org.gradle.jvmargs=-Xmx2048m -XX:MaxMetaspaceSize=512m -XX:+HeapDumpOnOutOfMemoryError

提示

用户级 ~/.gradle/gradle.properties 的设置会被项目级同名属性覆盖，适合存放 JVM 参数、凭证等机器级配置，避免每个项目重复声明。

注意

开启 org.gradle.configuration-cache=true 后，部分插件或自定义任务若不遵循配置缓存约束（如直接读取 project.properties、在配置阶段做文件 I/O），可能导致缓存失效或构建报错。建议先在本地稳定运行后再推广到 CI。

# Groovy DSL 与 Kotlin DSL 怎么选

维度 Groovy DSL Kotlin DSL
语法风格 接近脚本，灵活宽松 更接近 Java/Kotlin，类型安全
IDE 支持 基本补全 更好的自动补全与重构
学习曲线 对运维/脚本背景更友好 对 Java/Kotlin 开发者更友好
官方推荐 兼容旧项目 新项目首选（Gradle 8+ 默认）

提示

如果你是 Java/Kotlin 背景，或项目刚起步，优先选 Kotlin DSL；如果维护的是老项目或团队更熟悉 Groovy，保持 Groovy DSL 也完全可以。

# 五、核心机制：生命周期与任务图

Gradle 不是固定生命周期驱动，而是 任务图（Task Graph）驱动。

常见任务：

  • clean：清理构建产物
  • classes：编译主代码
  • test：执行测试
  • assemble：组装产物
  • check：执行质量校验（含测试）
  • build：完整构建（常用入口）

# 常用命令组合

./gradlew test
./gradlew build
./gradlew publishToMavenLocal

# 六、增量构建与任务状态

Gradle 在执行任务时会检查输入和输出是否变化，并在控制台输出中标注任务状态。理解这些状态有助于判断构建是否符合预期。

# 任务状态含义

状态 含义
无标注 任务正常执行
UP-TO-DATE 输入/输出均未变化，跳过执行
FROM-CACHE 输出从构建缓存（本地或远程）恢复
SKIPPED 满足了 onlyIf 条件导致跳过
NO-SOURCE 无源文件（如空测试目录），跳过

增量构建的前提是任务 声明了输入（inputs）和输出（outputs）。对于 compileJava、test、jar 等内置任务，Gradle 已内置声明。

# 常用诊断命令

# 强制重新执行某个任务（忽略 UP-TO-DATE / 缓存）
./gradlew test --rerun-tasks

# 查看某任务的详细输入输出信息
./gradlew :compileJava --info

# 启用构建缓存执行构建
./gradlew build --build-cache

# 七、核心机制：构建脚本关键结构

# plugins

声明构建能力来源，例如 java、application、maven-publish、checkstyle。

# repositories

声明依赖解析仓库来源，例如 mavenCentral() 或组织私服。

# dependencies

声明依赖与作用域（Configuration）：

  • implementation：主代码依赖，不对外暴露
  • api（java-library 插件）：主代码依赖，同时对外暴露给依赖方
  • compileOnly：仅编译，不打入产物
  • runtimeOnly：仅运行时，编译不可见
  • testImplementation：仅测试代码可见
  • testRuntimeOnly：仅测试运行时

# tasks

用于配置已有任务或注册新任务，承载测试参数、打包规则、代码生成等构建逻辑。

# 八、自定义 Task 基础

绝大多数场景直接配置内置 Task 即可，但遇到特殊构建步骤时，可以简单扩展。

# 方式一：脚本内联（适合简单一次性任务）

// build.gradle.kts
tasks.register("printBuildInfo") {
    group = "custom"
    description = "打印构建版本信息"
    doLast {
        println("Project: ${project.name}, version: ${project.version}")
    }
}

# 方式二：继承 DefaultTask（适合有输入输出的可缓存任务）

import org.gradle.api.DefaultTask
import org.gradle.api.provider.Property
import org.gradle.api.tasks.Input
import org.gradle.api.tasks.TaskAction

abstract class VersionBumpTask : DefaultTask() {
    @get:Input
    abstract val newVersion: Property<String>

    @TaskAction
    fun run() {
        println("Bumping version to ${newVersion.get()}")
    }
}

tasks.register<VersionBumpTask>("bumpVersion") {
    group = "release"
    newVersion.set("2.0.0")
}

# 冲突治理

当出现多版本竞争时，可通过以下方式约束：

  • 平台依赖（BOM）
  • constraints 明确版本
  • 必要时通过 exclude 排除不需要的传递依赖

当团队希望构建结果更可复现时，可开启依赖锁定（两种 DSL 语法相同）：

dependencyLocking {
    lockAllConfigurations()

Version Catalog 是 Gradle 7.0+ 推荐的统一依赖版本管理方式，通过 gradle/libs.versions.toml 集中声明版本、依赖库和插件。

文件位置：

gradle/

[bundles]

[plugins]

# 十三、排错与诊断（高频问题）

# 依赖版本异常

处理建议：

  • 执行 ./gradlew dependencies 查看依赖图
  • 执行 ./gradlew dependencyInsight --dependency <name> 定位冲突来源
  • 执行 ./gradlew dependencyInsight --dependency <name> --configuration runtimeClasspath 精确到目标配置

# 本地可过、CI 失败

常见原因：JDK、Gradle、缓存、仓库配置不一致。

处理建议：

  • 使用 Wrapper 固定版本
  • 固定 CI 的 JDK 版本（结合 Toolchain）
  • 固定仓库入口与凭证注入方式

# 任务行为不一致

常见原因：插件版本漂移、任务覆盖顺序不当。

处理建议：

  • 显式锁定插件版本
  • 将任务定制集中到根工程统一管理

# 十四、迁移专题：Maven 到 Gradle

# GAV 与 scope 映射

  • groupId:artifactId:version → 'group:artifact:version'
  • compile → implementation / api
  • provided → compileOnly
  • runtime → runtimeOnly
  • test → testImplementation

# 多模块迁移

  • Maven <modules> → Gradle include
  • 父 POM 统一管理 → 根工程统一配置

# 低风险迁移路径

  1. 双轨并行（Maven 与 Gradle 共存）
  2. 先打通最小闭环（编译、测试、打包、本地发布）
  3. 做产物一致性核验
  4. CI 灰度切换并保留回滚窗口

# 十五、工程化基线与扩展阅读

# 基线清单

  • 统一使用 Wrapper，锁定 Gradle 版本
  • 固定 JDK 与插件版本（推荐使用 Toolchain）
  • 依赖治理优先使用 BOM/约束/Version Catalog
  • 发布凭证外置（环境变量或 ~/.gradle/gradle.properties）
  • CI 启用构建缓存（org.gradle.caching=true）
  • CI 统一任务入口（优先 ./gradlew build）
  • 增量构建异常时通过 --info 排查输入输出声明
