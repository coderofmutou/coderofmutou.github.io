Maven 依赖管理与项目构建

# Maven 依赖管理与项目构建

# 一、概念与定位：Maven 解决的问题

Maven 的核心价值在两件事：

  1. 依赖管理：自动下载、传递依赖、版本统一、冲突裁决
  2. 构建管理：统一编译、测试、打包、安装、部署流程

Maven 可视作"项目构建与依赖的标准操作系统"。

提示

在团队协作中，Maven 真正的价值不是"方便下载 jar"，而是"让每个人、每台机器、每条 CI 流水线执行相同构建动作并得到可复现结果"。

# 二、环境准备与基础配置（Windows）

# 前置条件

  • JDK 17+（建议长期支持版本）
  • 正确设置 JAVA_HOME
  • mvn -v 能输出版本信息

# 建议检查项

mvn -v
1

期望输出至少包含：

  • Apache Maven 版本
  • Java 版本
  • 操作系统信息

# settings.xml 常见配置

通常在 Maven 安装目录下 conf/settings.xml 或用户目录下 .m2/settings.xml 中进行配置。

# 本地仓库路径

<localRepository>D:/maven-repository</localRepository>

        <name>aliyun maven</name>
        <url>https://maven.aliyun.com/repository/public</url>
        <mirrorOf>central</mirrorOf>

# 三、Maven Wrapper（mvnw）

Maven Wrapper（mvnw）的作用是在项目内锁定 Maven 版本，确保任何人/任何机器/任何 CI 环境都使用相同的 Maven 版本构建，无需机器预先安装指定版本。

# 生成 Wrapper

# 在项目根目录下执行（需要本机已有 Maven）
mvn wrapper:wrapper -Dmaven=3.9.9
1
2

│   └── wrapper/

├── mvnw                               # Unix/Linux/macOS 执行脚本
├── mvnw.cmd                           # Windows 执行脚本
└── pom.xml

# 四、最小工程实践：从创建到运行

# 项目坐标（GAV）

Maven 项目由三个核心标识组成：

  • groupId：组织/公司维度
  • artifactId：模块名称
  • version：版本号

完整坐标形式：groupId:artifactId:version

# GAV 命名约定（团队级）

在多人协作场景下，建议约定 GAV 命名规则，降低仓库污染与依赖歧义：

  • groupId：建议使用反域名结构，如 com.company.business
  • artifactId：建议使用"产品线-模块"语义，如 order-service-api
  • version：建议遵循 主版本.次版本.修订号（如 1.2.3）

提示

命名规则的核心目标是"可检索、可区分、可演进"，而不是追求形式复杂度。

# 最小可用 POM

<project xmlns="http://maven.apache.org/POM/4.0.0"
                 xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
                 xsi:schemaLocation="http://maven.apache.org/POM/4.0.0 https://maven.apache.org/xsd/maven-4.0.0.xsd">

        <modelVersion>4.0.0</modelVersion>

        <groupId>com.example</groupId>
        <artifactId>demo-maven</artifactId>
        <version>1.0.0-SNAPSHOT</version>

        <properties>
                <maven.compiler.release>17</maven.compiler.release>
                <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
                <project.reporting.outputEncoding>UTF-8</project.reporting.outputEncoding>
        </properties>

</project>

# 五、核心机制：生命周期与命令

Maven 内置三套生命周期：

  • clean
  • default
  • site

日常开发最常用的是 default 生命周期中的这几步：

阶段 作用
compile 编译主代码
test 运行单元测试
package 打包（jar/war）
verify 验证（含额外检查）
install 安装到本地仓库
deploy 部署到远程仓库

# 常用命令组合

mvn clean
mvn clean test
mvn clean package
mvn clean install
mvn clean deploy

# 生命周期、命令、插件的关系

Maven 构建过程可以理解为三层映射：

  1. 生命周期（Lifecycle）定义阶段序列
  2. 命令（Phase）触发对应阶段及其前置阶段
  3. 插件（Plugin Goal）最终完成实际动作

可简化为：Lifecycle -> Phase -> Plugin Goal。

例如执行 mvn package 时，本质上会触发 package 及其前置阶段， 并由编译、测试、打包相关插件完成具体工作。

提示

执行任意阶段会自动触发其所有前置阶段。

        <groupId>org.junit.jupiter</groupId>

</dependencies>

常见作用域：

  • compile：默认，编译/测试/运行都可见
  • test：仅测试可见
  • provided：编译与测试可见，运行由容器提供
  • runtime：运行与测试可见，编译不可见

补充作用域（高频于工程治理）：

  • system：与 provided 类似，但依赖来自本机路径；可移植性差，通常不建议使用
  • import：仅用于在 dependencyManagement 中导入 BOM（type=pom）

# 七、工程实践：依赖治理（传递、冲突、BOM）

# 传递依赖

如果 A 依赖 B，B 依赖 C，那么 A 默认会拿到 C（传递依赖）。

该机制提升了复用效率，但也可能导致依赖树快速膨胀。

# 冲突裁决规则

Maven 默认优先：

  1. 路径更短的版本（nearest）
  2. 路径同深度时，先声明优先

# 用 dependencyManagement 统一版本

<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-dependencies</artifactId>
            <version>3.3.1</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>

# 八、工程实践：Profiles 构建环境切换

Profiles 让同一套 POM 在不同环境下有不同构建行为，适合开发/测试/生产的差异化配置。

# 基础示例

<profiles>
    <profile>
        <id>dev</id>
        <activation>
            <activeByDefault>true</activeByDefault>
        </activation>
        <properties>
            <env>dev</env>
            <log.level>DEBUG</log.level>
        </properties>
    </profile>

    <profile>
        <id>prod</id>
        <properties>
            <env>prod</env>
            <log.level>WARN</log.level>
        </properties>
        <build>
            <plugins>
                <!-- 生产环境额外生成 javadoc 包 -->
                <plugin>
                    <groupId>org.apache.maven.plugins</groupId>
                    <artifactId>maven-javadoc-plugin</artifactId>
                    <executions>
                        <execution>
                            <goals><goal>jar</goal></goals>
                        </execution>
                    </executions>
                </plugin>
            </plugins>
        </build>
    </profile>
</profiles>

# Profile 定义位置与用途

定义位置 适用场景
pom.xml（<profiles>） 工程级差异，可移植，推荐优先
用户目录 .m2/settings.xml 机器级，适合包含账号密码等本机信息
全局 conf/settings.xml 全局默认，适合统一镜像/代理配置

提示

Profile 中的配置是对基础 POM 的 叠加 而非替换。激活多个 Profile 时，各 Profile 中的配置会依次合并。

注意

避免在 Profile 里做"大换血"——例如整体替换依赖列表——会导致构建行为难以追踪。Profile 的粒度越细越好。

# 九、工程实践：插件治理

推荐在父工程通过 pluginManagement 锁定插件版本：

<build>
    <pluginManagement>
        <plugins>
            <plugin>
                <groupId>org.apache.maven.plugins</groupId>
                <artifactId>maven-compiler-plugin</artifactId>
                <version>3.13.0</version>
            </plugin>
            <plugin>
                <groupId>org.apache.maven.plugins</groupId>
                <artifactId>maven-surefire-plugin</artifactId>
                <version>3.2.5</version>
            </plugin>
        </plugins>
    </pluginManagement>
</build>

maven-resources-plugin 处理 src/main/resources 资源文件，控制变量替换（filtering） 见下方"资源打包补充"
maven-dependency-plugin 提供 copy-dependencies 等目标，复制依赖 jar 到指定目录 <outputDirectory>
maven-shade-plugin 将依赖解压合并进单一可执行 jar（uber jar），支持按策略合并同名资源 <transformers>
maven-assembly-plugin 按描述符（内置或自定义）打包归档，可同时产出多种格式 <descriptorRefs> / <descriptor>
maven-javadoc-plugin 打包 API 文档 生成 -javadoc.jar
maven-checkstyle-plugin 代码风格检查 指定 checkstyle 规则文件
spotbugs-maven-plugin 静态缺陷扫描 绑定 verify 阶段

# 打包合并依赖：uber jar 与自定义 assembly 描述符

当需要产出一个"依赖全部合并在一起"的单一可执行 jar（通常称为 uber jar / fat jar，注意这与 Spring Boot 的可执行 jar 结构不同，后者是嵌套读取而非解压合并），常用 maven-shade-plugin：

<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-shade-plugin</artifactId>
    <version>3.5.1</version>
    <executions>
        <execution>
            <phase>package</phase>
            <goals><goal>shade</goal></goals>
            <configuration>
                <transformers>
                    <!-- 按行合并 META-INF/services/*，而非让后写入的覆盖先写入的 -->
                    <transformer implementation="org.apache.maven.plugins.shade.resource.ServicesResourceTransformer"/>
                    <transformer implementation="org.apache.maven.plugins.shade.resource.ManifestResourceTransformer">
                        <mainClass>com.example.Application</mainClass>
                    </transformer>
                </transformers>
            </configuration>
        </execution>
    </executions>
</plugin>

# 资源打包补充：多组 resources 与 filtering

maven-resources-plugin 默认只处理 src/main/resources 下的全部文件，原样复制到 target/classes。若需要更精细的控制——例如部分文件需进行变量替换（filtering），部分文件不需要，或按构建参数动态选择需打入的文件——可在 <build> 下声明多组 <resource>，每组独立配置 directory/includes/excludes/filtering/targetPath：

<build>
    <resources>
        <!-- 第一组：静态前端资源，重定向到 static 目录，不做变量替换 -->
        <resource>
            <directory>src/main/webapp</directory>
            <targetPath>static</targetPath>
            <includes>
                <include>**/**</include>
            </includes>
            <excludes>
                <exclude>**/*.jsp</exclude>
            </excludes>
        </resource>
        <!-- 第二组：常规配置文件，开启filtering做变量替换，但排除需要单独处理的部分 -->
        <resource>
            <directory>src/main/resources</directory>
            <includes>
                <include>**/**</include>
            </includes>
            <excludes>
                <exclude>profiles/**/*.*</exclude>
                <exclude>templates/**</exclude>
            </excludes>
            <filtering>true</filtering>
        </resource>
        <!-- 第三组：模板文件本身含有与filtering语法冲突的占位符，单独关闭filtering -->
        <resource>
            <directory>src/main/resources</directory>
            <includes>
                <include>templates/**</include>
            </includes>
            <filtering>false</filtering>
        </resource>
        <!-- 第四组：按构建参数（如 -Denv=xxx 传入的属性）动态选择需打入哪个子目录下的文件 -->
        <resource>
            <directory>src/main/resources</directory>
            <includes>
                <include>profiles/${env}/*.xml</include>
            </includes>
            <filtering>true</filtering>
        </resource>
    </resources>
</build>

# 十、工程实践：编写自定义 Maven 插件

标准插件覆盖不到的构建需求——例如构建期动态生成配置文件、按参数拉取额外资源——可通过编写自定义 Maven 插件解决。自定义插件本质上是一个实现特定接口的 Java 类（Mojo），由 Maven 在指定生命周期阶段调用执行。

# 编写步骤

  1. 新建一个独立的 Maven 项目，packaging 设为 maven-plugin，引入 maven-plugin-api 和 maven-plugin-annotations 两个依赖
  2. 定义一个继承 AbstractMojo 的类，用 @Mojo(name = "目标名", defaultPhase = LifecyclePhase.PREPARE_PACKAGE) 声明目标（goal）名称及其默认绑定的生命周期阶段
  3. 需要从外部 pom 的 <configuration> 接收参数时，用 @Parameter 注解声明对应字段；具体处理逻辑写在 execute() 方法中
@Mojo(name = "make-plugins", defaultPhase = LifecyclePhase.PREPARE_PACKAGE)
public class MakePluginsMojo extends AbstractMojo {

    @Parameter(property = "pid", required = true)
    private String pid;

    @Override
    public void execute() throws MojoExecutionException {
        // 具体处理逻辑：按 pid 参数生成/拉取所需资源，不在此展开
    }
}

# 适用场景与选型

企业级项目中常见此类自定义插件，例如按某个构建参数（如 ${pid} 这类客户/环境标识）在构建期动态处理定制化资源。这类插件的目标名和内部逻辑因项目而异，把握上述骨架即可理解其基本机制，具体业务逻辑不必逐一展开。

# 十一、工程实践：Enforcer 构建约束

maven-enforcer-plugin 用于在构建阶段主动检查项目健康度，是企业级 Java 项目的常见配置。

# 常见规则示例

<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-enforcer-plugin</artifactId>
    <version>3.5.0</version>
    <executions>
        <execution>
            <id>enforce</id>
            <goals><goal>enforce</goal></goals>
            <configuration>
                <rules>
                    <!-- 依赖版本收敛：同一制品在依赖树中只允许出现一个版本 -->
                    <dependencyConvergence/>

                    <!-- 要求所有插件都显式声明版本，防止版本漂移 -->
                    <requirePluginVersions>
                        <banLatest>true</banLatest>
                        <banSnapshots>true</banSnapshots>
                    </requirePluginVersions>

                    <!-- 约束 JDK 版本范围 -->
                    <requireJavaVersion>
                        <version>[17,)</version>
                    </requireJavaVersion>

                    <!-- 约束 Maven 版本范围 -->
                    <requireMavenVersion>
                        <version>[3.8,)</version>
                    </requireMavenVersion>
                </rules>
            </configuration>
        </execution>
    </executions>
</plugin>

# 常用规则速查

规则 作用
dependencyConvergence 禁止依赖树中同一坐标出现多个版本
requirePluginVersions 要求所有插件显式声明版本
requireJavaVersion 约束构建所用 JDK 版本
requireMavenVersion 约束构建所用 Maven 版本
requireUpperBoundDeps 禁止传递依赖引入高于声明版本的制品
bannedDependencies 禁止特定依赖进入构建

注意

dependencyConvergence 在引入初期往往会暴露大量隐藏冲突，建议先在本地用 mvn dependency:tree -Dverbose 梳理，逐步通过 dependencyManagement 解决冲突，再在 CI 中开启强制校验。

# SNAPSHOT 与 Release 语义

Maven 用版本号后缀区分两种制品状态：

  • SNAPSHOT 版本：版本号以 -SNAPSHOT 结尾，表示仍在开发中，允许覆盖发布。每次拉取时 Maven 会按一定策略检查远程仓库是否有更新。
  • Release 版本：版本号不带 -SNAPSHOT（如 1.2.3），一旦发布即不可变，任何修改都必须升级版本号重新发布。

私服仓库通常也按这两类划分：

仓库类型 用途
maven-snapshots 接收 SNAPSHOT 版本，允许重复覆盖
maven-releases 接收 Release 版本，通常禁止覆盖

注意

不要把未稳定的 SNAPSHOT 版本带入生产发布；同时，Release 版本一旦发布就不要再修改同名制品，否则会导致依赖缓存不一致。

# 十四、排错与诊断（高频问题）

# 依赖下载失败 / 一直失败不重试

可能原因：本地仓库残留 .lastUpdated。

处理思路：

  1. 先确认坐标准确
  2. 再确认网络和镜像
  3. 最后清理目标依赖目录中的 .lastUpdated 文件

# 编译版本报错

检查：

  • mvn -v 的 Java 版本
  • maven.compiler.release 或 source/target 是否一致

# 类冲突 / NoSuchMethodError

大概率是依赖树冲突导致版本错配。

# 十五、工程化基线清单

  • 项目使用 Maven Wrapper（mvnw），锁定 Maven 版本
  • 父工程统一依赖和插件版本
  • 关键依赖走 BOM 或 dependencyManagement
  • CI 中启用 maven-enforcer-plugin，约束 JDK/Maven 版本与依赖收敛
  • 多环境差异通过 Profiles 管理，避免维护多套 POM
  • 避免 system 作用域，优先上私服
  • 不要把业务逻辑写进构建脚本
  • CI 使用固定 Maven 与 JDK 版本
  • 发布前至少执行 ./mvnw clean verify
